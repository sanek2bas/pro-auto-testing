Unit_Testing_Private_Methods
Sometimes private methods prove too complex, and testing them as part of the 
observable behavior doesn't provide sufficient coverage. 
The GenerateDescription() method is very simple: it returns a general description
of the order. However, it uses a private GetPrice() method, which is much 
more complex: it contains important business logic and requires thorough testing. 
This logic is the missing abstraction. Instead of exposing the GetPrice method, 
make this abstraction explicit.

Revealing_Private_State
Another common anti-pattern is exposing private state for unit testing. 
The rule here is the same as with private methods: don't expose state that
you would otherwise prefer to keep private—test only observable behavior.
How do I test the Promote() method? This method has the side effect of changing
the _status field, but the field itself is private and therefore inaccessible 
in tests. We need to look at how the production code uses this class. 
In this particular example, the customer status isn't important to the 
production code; otherwise, this field would be public. The only thing 
the production code cares about is the discount the customer will receive 
after upgrading their status. Therefore, this is what needs to be tested.

Domain_Knowledge_Leakage_Into_Tests
Leaking domain knowledge into tests is another common anti-pattern. 
It typically occurs in tests covering complex algorithms (class Calculator).
WrongCalculatorTests - these tests duplicate the implementation of the algorithm
from the production code.
RightCalculatorTests - it may seem wrong at first, but coding the expected result 
is a good practice in unit testing. It's important to calculate this result in 
advance using something other than the code of the system under test 
(ideally with the help of a subject matter expert). Of course, this only applies
to fairly complex algorithms (we're all experts at adding two numbers). 
Alternatively, if you're refactoring a legacy application, you can obtain these 
results using the old code and then use them as expected values ​​in your tests.
