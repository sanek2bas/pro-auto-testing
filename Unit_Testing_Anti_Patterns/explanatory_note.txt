Unit_Testing_Private_Methods
Sometimes private methods prove too complex, and testing them as part of the 
observable behavior doesn't provide sufficient coverage. 
The GenerateDescription() method is very simple: it returns a general description
of the order. However, it uses a private GetPrice() method, which is much 
more complex: it contains important business logic and requires thorough testing. 
This logic is the missing abstraction. Instead of exposing the GetPrice method, 
make this abstraction explicit.

Revealing_Private_State
Another common anti-pattern is exposing private state for unit testing. 
The rule here is the same as with private methods: don't expose state that
you would otherwise prefer to keep private—test only observable behavior.
How do I test the Promote() method? This method has the side effect of changing
the _status field, but the field itself is private and therefore inaccessible 
in tests. We need to look at how the production code uses this class. 
In this particular example, the customer status isn't important to the 
production code; otherwise, this field would be public. The only thing 
the production code cares about is the discount the customer will receive 
after upgrading their status. Therefore, this is what needs to be tested.

Domain_Knowledge_Leakage_Into_Tests
Leaking domain knowledge into tests is another common anti-pattern. 
It typically occurs in tests covering complex algorithms (class Calculator).
WrongCalculatorTests - these tests duplicate the implementation of the algorithm
from the production code.
RightCalculatorTests - it may seem wrong at first, but coding the expected result 
is a good practice in unit testing. It's important to calculate this result in 
advance using something other than the code of the system under test 
(ideally with the help of a subject matter expert). Of course, this only applies
to fairly complex algorithms (we're all experts at adding two numbers). 
Alternatively, if you're refactoring a legacy application, you can obtain these 
results using the old code and then use them as expected values ​​in your tests.

Code_Pollution
This is adding working code that is only needed for testing. In this example, 
the Logger class is passed a parameter in its constructor that indicates whether
the class is running in production. If so, a message is written to a file; 
otherwise, nothing happens. This switch allows you to disable logging during 
test runs. The problem with code pollution is that test code gets mixed in with 
production code, increasing the cost of maintaining the latter. To avoid this
anti-pattern, separate test code from production code. In the Logger example, 
add the ILogger interface and create two implementations: a real one for 
production and a dummy one for testing purposes. Then, refactor the Controller 
class so that it receives the interface instead of a concrete class.

Mocking_Concrete_Classes
You can replace concrete classes with mocks and thus preserve some of the 
functionality of the original classes, which can sometimes be useful. 
However, this alternative has a significant drawback: it violates the 
single responsibility principle.
How can I test this controller? It can't be passed a real instance of 
StatisticsCalculator because that instance accesses an unmanaged out-of-process 
dependency. The unmanaged dependency should be replaced with a stub.
At the same time, completely replacing StatisticsCalculator is also undesirable. 
This class contains important computational functionality that should remain 
unchanged. One way to deal with this dilemma is to mock the StatisticsCalculator
class and override only the GetDeliveries() nethod, which can be done by declaring
this method virtual (WrongCustomerControllerTests). Setting CallBase = true allows 
the mock to retain the base class's behavior unless it's explicitly overridden. 
This approach allows you to replace only part of the class while leaving the 
rest as is. This is an anti-pattern.
StatisticsCalculator combines two unrelated responsibilities: interacting with 
an unmanaged dependency and calculating statistics. The domain logic is in the 
Calculate() method. GetDeliveries() only collects input for this logic. 
Instead of mocking StatisticsCalculator, split the class into two (StatisticsCalculator2
and CustomerController2). The responsibility for interacting with the unmanaged
dependency has been transferred to DeliveryGateway. Note that this gateway now 
operates on an interface that can be used for mocking instead of a concrete class.
