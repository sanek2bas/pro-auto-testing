The general recommendation for integration testing is to cover
the longest positive path and any edge cases that cannot be 
handled by unit tests. The longest positive it the one that
passes throught all out-of-process dependencies.  

Regarding edge cases not tested by unit tests, there is only one: 
a script whose address cannot be changed. However, testing this scenario

Before writing integration tests, you need to classify the two out-of-process 
dependencies and decide which one should be tested directly and which should be 
replaced with a mock. The application database is a managed dependency because 
no other system can access it. Therefore, you should use a real instance of it. 
The integration test should:
-insert user and company data into the database;
-execute an email update scenario against this database;
-check the state of the database.
On the other hand, the message bus is an unmanaged dependency—it is needed solely 
to ensure interaction with other systems. The integration test will replace the 
message bus with a mock and then test the interactions between the controller and this mock.

It's crucial to check the database state independently of the data used as input parameters. 
To do this, the integration test separately requests user and company information 
in the assertion section, creates new userFromDb and companyFromDb instances, 
and only then checks their state. This approach ensures that the test verifies 
both writing and reading from the database, thus providing maximum protection against bugs. 
The reading itself should be performed using the same code used in the controller's internal implementation; 
in this example, this is the code for the Database, UserFactory, and CompanyFactory classes.

Don't write interfaces for out-of-process dependencies unless you need to replace them based on a mock.
Since mocks should only be used for unmanaged dependencies, this recommendation boils down to this: 
use interfaces only for unmanaged dependencies. Managed dependencies should still be explicitly injected 
into the controller, but use specific classes for this purpose.

Interactions with out-of-process dependencies should be considered based on whether these interactions
are part of the application's observable behavior or an implementation detail. 
Logging is no different in this regard. Replace logging functionality with a mock if the
logs are accessible to non-programmers; otherwise, there's no need to test it.

This test replaces two unmanaged dependencies with mocks: IMessageBus and IDomainLogger.
Let's look at IMessageBus first. IDomainLogger will be discussed later in this chapter.
When using mocks, always follow this principle: test interactions with unmanaged dependencies 
at the very edges of your system.

An unmanaged dependency goes through several stages before exiting your application.
Choose the last of these stages. This is the best way to ensure backward compatibility 
with external systems, which is the purpose of mocks. So IMessageBus was replace with IBus.

A Spy is type of test stub that serves the same purpose as a mock. The only difference is that
spies are written by hand, while mocks are created using mocking frameworks. For classes
located near the boundaries of a system, spies are preferable to mocks, Spies help
reuse code during the verification phase thereby reducing the size and improving
the readability of tests.

BusSpy and MessageBus are wrappers for IBus. However, there is a fundamental difference between them:
BusSpy is part of the test code, while MessageBus is part of the production code. 
This is an important distinction, because when using assertions in tests, you shouldn't rely on the production code.
Your tests are the code's auditors. A good auditor doesn't simply take the word of the 
person being audited at face value; they check and double-check everything.

The rule that mocks should only be used in integration tests, not unit tests, follows from the 
fundamental principle: separation of business logic and coordination. Your code should either interact 
with out-of-process dependencies or be complex—but not both. This principle leads to the formation of two layers: 
the domain model layer (which is responsible for complexity) and the controller layer (which is responsible for interactions).
Domain model tests are categorized as unit tests; tests covering the controller are integration tests. 
Since mocks are intended only for unmanaged dependencies, and controllers are the only code that works 
with such dependencies, mocks should only be used when testing controllers—in integration tests.

Not all unmanaged dependencies require the same level of backward compatibility. 
If the exact structure of a message is unimportant and you only want to verify the existence of the message 
and the information it contains, you can ignore the recommendation to verify interactions with unmanaged dependencies 
at the boundaries of your system. A typical example of this is logging.
